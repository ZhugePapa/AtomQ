# 知识卡片内容规范与 AI 提示词

本文档用于指导“高级-信息系统项目管理师”知识卡片的生成、审校和修复。目标不是把 PDF 原文搬进 App，而是把考试资料整理成适合移动端碎片学习、遮挡记忆、刷题回看和长期复习的内容资产。

## 一、内容目标

一张好的知识卡片应该同时满足三个目标：

1. **讲得清楚**：用户第一次看能理解概念、边界、用途和易混点。
2. **背得起来**：核心词、顺序、角色、公式、分类能被快速遮挡记忆。
3. **考得上题**：能对应选择题常见问法，帮助用户区分相近概念。

内容应优先服务“3 分钟碎片学习”和“答题时快速回看”，避免写成教材长文。

## 二、文件与数据规范

1. 每张卡片由同名 `.json` 和 `.md` 组成，文件名格式为：

   ```text
   chapter_id_section_id_point_id
   ```

   示例：`ch_01_sec_02_015.json` 与 `ch_01_sec_02_015.md`。

2. `subject_id` 固定为 `high_itpmp`。

3. `chapter_id` 使用 `ch_01` 至 `ch_23`。

4. `section_id` 使用 `sec_01`、`sec_02` 这种章内编号，不再包含章节号。

5. `point_id` 使用三位数字，如 `001`、`015`，不再包含 `kp` 或章节号。

6. JSON 负责程序消费字段：标题、类型、难度、标签、路径、关联关系、`key_points`、`mnemonics`、来源。

7. Markdown 负责正文阅读体验：解释、表格、列表、重点遮挡、易混点。

8. `source_refs` 建议统一使用：

   ```json
   {
     "source_id": "pdf_2026_tricolor_high",
     "page_label": "17,18"
   }
   ```

## 三、Markdown 正文规范

1. 正文标题从 `###` 开始。正文内如需更低层级标题，依次使用 `####`，但不要滥用标题。

2. 常用结构：

   ```markdown
   ### 概述

   用 1-3 段讲清楚概念、边界和考试语境。

   ### 核心内容

   用表格、编号列表或短段落呈现分类、流程、角色、公式。

   ### 易混点

   写选择题最容易混的判断规则。
   ```

3. `==重要==` 只用于正文 Markdown 中的遮挡记忆内容，不用于 JSON 字段。

4. `==重要==` 标注应只包住核心词、核心短语、数字、顺序、角色称谓、公式结论，不要整段高亮。

5. 同一组并列答案如果都属于需要遮挡记忆的对象，应一致标注，例如“硬件、软件、数据库、网络……”这类枚举项不要只标一部分。

6. PDF 红字通常转成正文中的 `==重要==`，但要经过人工判断：红字如果是整句说明，应只保留其中真正需要背的关键词。

7. 不保留 OCR 噪声、页脚、水印、资料广告、残缺编号、孤立短词和无法解释的碎片。

8. 表格用于分类、对比、流程输入输出、角色责任；不要把本来适合一句话讲清楚的内容硬做表格。

9. 卡片应保持“原子化”：一张卡只讲一个概念、一个流程、一个分类或一个易混组合。

## 四、JSON 字段内容规范

### key_points

`key_points` 是独立 UI 卡片中的“考试抓手”，不是 Markdown 正文的重复摘录。

1. 可以为空。没有明确考试抓手时，宁可留空，不要制造记忆负担。
2. 应使用 Markdown 语法字符串，允许列表、表格、加粗。
3. JSON 中不能使用 `==...==`，需要强调时使用 `**...**`。
4. 内容应优先包含：
   - 高频选择题判断规则
   - 易错角色/顺序/范围
   - 数字、阶段、分类、公式
   - 与相近概念的区分
5. 不应包含：
   - 正文已经完整解释、且没有额外记忆价值的句子
   - 空泛鼓励语
   - 太长的原文堆砌

### mnemonics

`mnemonics` 是记忆口诀，也可以为空。

1. 只在确实能降低记忆成本时填写。
2. 口诀必须可回推原始知识点，不能为了押韵牺牲准确性。
3. 对复杂流程、长枚举、固定顺序适合写口诀；对单一概念、理解型内容不必强行写。
4. JSON 中强调使用 `**...**`，不用 `==...==`。

## 五、学习体验标准

1. **先理解后背诵**：卡片先讲“它是什么/为什么”，再给“怎么记/怎么考”。
2. **短而完整**：不要薄到只剩一个词，也不要厚到像教材小节。
3. **显式区分易混概念**：例如“龙头 vs 核心”“数字产业化 vs 产业数字化”“信息基础设施 vs 融合基础设施”。
4. **按题目问法组织抓手**：看到什么关键词选什么，哪些词是陷阱。
5. **保留考试资料重点，但不迷信原文排版**：PDF 的红字、表格、编号是线索，不是最终形态。
6. **来源可追踪**：每张卡片应能回到 PDF 页码；涉及政策或标准的内容，优先使用官方来源校验。

## 六、质量检查清单

生成或修复后至少检查以下问题：

1. 是否有 OCR 残留：断字、乱码、孤立编号、页码、水印、广告。
2. 是否有错误编号：`1.`、`2.` 嵌进段落，或列表层级断裂。
3. 是否有整段 `==...==`：高亮应缩小到核心词。
4. JSON 中是否残留 `==...==`。
5. `key_points` 是否和正文“易混点/考试抓手”重复。
6. `mnemonics` 是否牵强或无法回推。
7. 表格是否移动端可读，是否需要拆成更小的表。
8. 标题是否准确，是否出现“核心内容”“章节考点”等无信息标题。
9. 来源页码是否缺失或格式不一致。
10. 卡片是否真的帮助用户做题，而不是只把资料换一种排版。

## 七、AI 生成提示词

```text
你是“高级-信息系统项目管理师”软考知识卡片内容编辑。请根据输入的 PDF OCR 文本、红字标记、章节/小节信息和既有数据规范，生成适合移动端学习的知识卡片数据。

任务目标：
1. 把资料整理成“章 -> 节 -> 知识卡片”的结构。
2. 每张卡片只讲一个清晰知识点，适合 3 分钟内阅读。
3. Markdown 正文用于讲解和遮挡记忆；JSON 用于程序消费和独立 UI 卡片展示。
4. 不照抄 PDF，不保留 OCR 噪声，要重组为易理解、易背诵、易刷题回看的内容。

命名规则：
- subject_id 固定为 high_itpmp。
- chapter_id 使用 ch_01 这种格式。
- section_id 使用 sec_01 这种格式，不包含章节号。
- point_id 使用 001 这种三位数字，不包含 kp 或章节号。
- 文件名为 chapter_id_section_id_point_id，例如 ch_01_sec_02_015。

Markdown 正文规则：
- 标题从 ### 开始。
- 常用结构为：### 概述、### 核心内容、### 易混点。
- PDF 红字表示重点，但不要整段高亮；只把需要用户遮挡记忆的核心词、数字、顺序、角色、公式用 ==重要== 标注。
- 同一组并列核心答案要一致标注。
- 可以使用表格、编号列表和短段落；表格要适合移动端阅读。
- 删除页码、水印、广告、残缺编号、乱码和无意义碎片。

JSON 字段规则：
- key_points 和 mnemonics 使用 Markdown 字符串，可以为空。
- key_points 是“考试抓手”，应写高频判断、易错点、顺序、分类、公式或概念边界，不要重复正文。
- mnemonics 只有在能明显降低记忆难度时填写，不要强行编口诀。
- key_points 和 mnemonics 中不能使用 ==...==，强调统一用 **...**。
- tags 控制在 3-6 个，包含章节主题、核心概念、易检索关键词。
- source_refs 使用 source_id + page_label，保留可追踪页码。

审校要求：
1. 先列出本章识别到的 section 数和卡片数。
2. 生成后逐卡检查：标题是否准确、正文是否完整、重点是否标对、key_points 是否有学习价值、mnemonics 是否必要。
3. 对政策、标准、PMBOK/项目管理概念类内容，优先用官方来源校验，不确定时标记为 ai_draft 并写出待复核点。
4. 最终输出 JSON 与 Markdown 内容，并给出本章遗留风险清单。
```
