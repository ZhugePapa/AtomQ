# Codex 执行规范（AtomQ）

## 1. 目标
- 本文档用于约束“你给我下指令后，我该如何执行”。
- 它是总规范，变量对齐只是其中一个模块，后续可持续补充。
- 所有执行结果以“可复现、可审计、可回滚”为标准。

## 2. 全局执行原则
- 不猜测需求：先精确理解指令目标、范围、输出形式。
- 不跳步骤：先采集事实，再下结论，再改动，再校验。
- 不以“看起来对”代替“证据对”：必须给出可追溯依据。
- 不做隐式大改：任何非显然改动要先说明影响。
- 不留黑箱：每次执行要输出关键过程、结论、风险。

## 3. 标准执行流程（默认）
1. 复述任务边界：明确“做什么、不做什么”。
2. 获取上下文：读取相关文件、节点、变量、日志、报错。
3. 建立检查清单：列出必检项与通过标准。
4. 执行改动：只改必要内容，避免无关改动。
5. 本地验证：编译/运行/对比/截图（按任务类型）。
6. 输出结果：改了什么、为什么、如何验证、剩余风险。

## 4. 输出要求
- 必须包含：
  - 改动文件路径
  - 关键改动点
  - 验证方法与结果
  - 未完成项（若有）
- 涉及对齐/审计任务时，必须包含差异清单（若存在）。

## 5. 风险控制
- 禁止未经确认的破坏性操作（删除大量文件、重置、回滚他人修改）。
- 发现上下游依赖冲突时先暂停并说明，再继续。
- 若工具结果不完整（截断/分页/缓存），必须补拉完整数据再结论。

## 6. 模块规范：Figma 对齐（当前已启用）

### 6.1 适用范围
- 所有 Figma 到代码的还原与校验任务（颜色、尺寸、布局、图标、变量）。

### 6.2 核心要求
- 不仅校验值，还要校验变量身份（变量名 + VariableID）。
- 不做局部抽查，必须对目标节点递归检查。
- Light/Dark 必须分别校验。

### 6.3 双重门禁
- 门禁 A（身份一致）：
  - 节点绑定变量与目标 Figma 变量一致。
- 门禁 B（模式一致）：
  - Light/Dark 值与 alias 关系一致。

### 6.4 必检项
- fill / stroke / text / effect 的变量绑定。
- SVG 图标是否存在硬编码 fallback 色。
- SVG 是否注入动态变量（如 `--stroke-0`）并绑定 token。

### 6.5 强制输出
- 差异清单（节点 ID、属性、Figma 变量、代码变量、修复动作）。
- 差异为 0 时给通过报告（检查数量 + 模式复核结果）。

### 6.6 数据源策略（强制）
- Figma 数据读取与审计默认且优先使用 Codex 官方 Figma 插件（MCP）链路。
- 禁止默认使用“用户 token + Figma REST API（curl / api.figma.com）”链路。
- 仅当你明确下达“允许使用 token API”的指令时，才可临时切换到 REST API。
- 若 MCP 与 token API 结果冲突，以 MCP 结果为准，并在报告中标注冲突点。

### 6.7 图标几何一致性防错（强制）
- 颜色变量对齐不代表视觉对齐，图标必须额外校验几何参数：`frame尺寸`、`viewBox尺寸`、`stroke-width`、`最终缩放倍率`。
- 若 SVG 存在缩放，必须计算渲染后描边：`renderedStroke = svgStrokeWidth * frameSize / viewBoxSize`，与 Figma 的实际描边一致才可通过。
- 对于小尺寸几何图形（如红点、圆环、细线、<12px 图标），优先使用 SwiftUI 原生图形（`Circle/Rectangle/Path`）实现，避免 WebView/SVG 缩放导致描边失真。
- 禁止仅凭“变量名一致”判定通过；必须同时通过“变量一致 + 几何一致”双检查。
- 若发现是几何问题，修复优先级高于颜色微调，避免误把几何误差当作变量错误处理。

## 7. 模块规范：代码改动
- 仅改动与任务直接相关文件。
- 变量命名与语义需和设计系统一致。
- 新增 token 时，注明用途并保持可追溯映射。
- 修改后必须至少完成一次编译校验（如环境允许）。

## 8. 模块规范：排障
- 先定位根因，再修复表象。
- 每个结论至少有一条事实证据（日志、节点数据、代码行）。
- 修复后必须回归验证同类问题点，防止“修一处坏一处”。

## 9. 文档维护机制
- 本文件是执行基线，后续新规则按模块追加。
- 每次出现漏检/误判，必须补充“防呆规则”到对应模块。
- 变更方式：
  - 写明“新增/修改/删除”的规则条目
  - 给出触发场景和预期收益
