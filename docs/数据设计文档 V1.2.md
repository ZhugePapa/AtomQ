## 1. 数据存放、读写与工程化方案

### 1.1 数据分层策略

整个 App 的数据分为两类，按**谁来消费**选择最合适的格式：

| 层        | 内容             | 格式           | 存储位置       | 读写方式            |
| :------- | :------------- | :----------- | :--------- | :-------------- |
| 静态 — 元数据 | 卡片属性、章节结构、题目索引 | **JSON**     | 本地沙盒 + CDN | 只读，`Codable` 解码 |
| 静态 — 正文  | 知识卡片正文内容       | **Markdown** | 本地沙盒 + CDN | 只读，MD 解析器渲染     |
| 动态数据     | 学习记录、错题、掌握度、任务 | **SQLite**   | 本地 + 云端同步  | 读写，GRDB / SQL   |

**设计原则**：
- **程序需查询筛选的数据**（标题、难度、类型、关联关系）→ JSON，`Codable` 一行解码，服务端可直接入库查询
- **人需编辑阅读的内容**（段落、表格、高亮、重点标注）→ Markdown，记事本就能改，Git diff 一目了然
- **频繁变动的数据**（学习记录、错题）→ SQLite，有完整的查询和事务支持

### 1.2 三库目录结构总览

```
软考刷题App 数据体系
│
├── 元数据库（JSON）        程序消费、服务端查询
├── 正文库（Markdown）      人工编辑、客户端渲染
└── 动态库（SQLite）        本地读写、云端同步
```

### 1.3 元数据库（JSON）

内容仓库根目录 → Git 版本控制 → CDN 分发。

```
知识点集合/
│
├── subject_index.json               # 科目总索引（<1KB）
│
├── chapters/
│   ├── ch_04/                       # 第 4 章：项目整合管理
│   │   ├── chapter_meta.json        #   章节元数据（<500B）
│   │   └── cards/
│   │       ├── kp_ch04_001.json     #   卡片元数据（<1KB）
│   │       ├── kp_ch04_002.json
│   │       ├── kp_ch04_003.json
│   │       └── ...
│   │
│   ├── ch_05/                       # 第 5 章：项目范围管理
│   │   ├── chapter_meta.json
│   │   └── cards/
│   │       └── ...
│   └── ...
│
└── questions/
    ├── ch_04_questions.json         # 第 4 章题目（含 difficulty、knowledge_point_ids）
    ├── ch_05_questions.json
    └── ...
```

| 文件 | 内容 | 大小 |
|:---|:---|:---|
| `subject_index.json` | 科目 ID、名称、章节列表（id + title + card_count） | < 1KB |
| `chapter_meta.json` | 章节 ID、标题、描述、排序号、子章节列表（V1.4） | < 1KB |
| `kp_*.json` | `point_id`、`title`、`card_type`、`difficulty`、`has_key_content`、`content_file`、关联关系、版权信息 | < 1KB |
| `ch_*_questions.json` | 该章节所有题目的数组，每题含 `knowledge_point_ids`、`difficulty`、题干选项答案解析 | 按题量 |

### 1.4 正文库（Markdown）

与元数据同目录，和 JSON 文件并肩存放，通过 `content_file` 字段关联：

```
知识点集合/
└── chapters/
    └── ch_04/
        └── cards/
            ├── kp_ch04_001.json     ← 元数据（content_file → kp_ch04_001.md）
            ├── kp_ch04_001.md       ← 正文
            ├── kp_ch04_002.json
            ├── kp_ch04_002.md
            └── ...
```

**示例 — 正文 Markdown**（只管内容表达和视觉标注）：

```markdown
### 概述

制定项目章程是编写一份正式批准项目并==授权项目经理==的文件的过程。

:::tip
**记忆口诀**：章程给名分，计划给方向。
:::

<!-- render:native caption:制定项目章程的输入输出 -->
| 输入 | 工具与技术 | 输出 |
|:---|:---|:---|
| 项目章程 | 专家判断 | 项目章程（签发） |
```

**为什么 JSON 和 MD 同目录？**
- 内容运营打开 `cards/` 目录，相邻的 json 和 md 双双可见
- 加载逻辑简单：`meta.content_file` 直接取文件名，拼接同目录路径即得 MD
- CDN 更新互不干扰：元数据修正只下发 JSON，正文修正只下发 MD

### 1.5 动态库（SQLite）

#### 1.5.1 iOS 客户端本地数据库

```
App 沙盒 ── Documents/
              └── data.db              ← SQLite 数据库（GRDB 管理）
                   │
                   ├── user_records    # 学习记录（含 answer_time_ms）
                   ├── wrong_book      # 错题本（含 next_review_at）
                   ├── mastery_scores  # 掌握度（0-100）
                   └── daily_tasks     # 每日计划任务
```

每个表带 `synced` 字段标记同步状态。

#### 1.5.2 服务端数据库

```
服务端 PostgreSQL
│
├── users                # 用户账号
├── user_records         # ← 客户端 sync 上来的学习记录
├── wrong_book           # ← sync + 服务端艾宾浩斯纠偏
├── mastery_scores       # ← sync
└── daily_tasks          # ← sync
```

#### 1.5.3 同步流向

```
iOS SQLite (data.db)
      │
      │  synced = 0 的行 → POST /api/sync → 逐条上传
      ▼
服务端 PostgreSQL
      │
      │  纠偏数据（next_review_at 校准）→ GET /api/sync/pull
      ▼
iOS SQLite  ← 写入纠偏值，标记 synced = 1
```

### 1.6 本地沙盒缓存 + Bundle 兜底

App 运行时，静态内容（JSON + MD）从 CDN 下载缓存到沙盒：

```
App 沙盒 ── Documents/
              ├── data.db              ← SQLite（动态数据）
              │
              └── cache/cards/         ← CDN 下载缓存（静态内容）
                  ├── subject_index.json
                  ├── ch_04/
                  │   ├── chapter_meta.json
                  │   └── cards/
                  │       ├── kp_ch04_001.json
                  │       ├── kp_ch04_001.md
                  │       └── ...
                  ├── ch_05/
                  │   └── ...
                  └── questions/
                      ├── ch_04_questions.json
                      └── ...
```

**Bundle 内置兜底数据**（App 首次安装无网络时使用）：

```
App Bundle ── Resources/
               └── default_cards/
                   ├── subject_index.json
                   └── ch_04/
                       └── cards/     ← 至少预置首章 JSON + MD
                           ├── kp_ch04_001.json
                           ├── kp_ch04_001.md
                           └── ...
```

**读取优先级**：

```
请求卡片 kp_ch04_001
        │
        ▼
1. Documents/cache/cards/ch_04/cards/kp_ch04_001.json  ← 本地缓存优先
        │
        ├─ 命中 → 返回元数据，content_file 指向同目录 .md
        │
        └─ 未命中 ↓
2. Bundle/Resources/default_cards/...                   ← 兜底数据
        │
        └─ 未命中 ↓
3. CDN 下载 → 写入 cache → 返回                           ← 首次加载
```

### 1.7 动态数据：SQLite 表结构

对应 PRD 第 12 节的数据对象定义及数据字典 V1.3 的用户对象，本地 SQLite 建表如下：

```sql
-- 用户账号（数据字典 V1.3 新增，PRD 12.1）
CREATE TABLE users (
    user_id       TEXT PRIMARY KEY,
    login_status  TEXT NOT NULL,  -- guest / login / vip
    subject_id    TEXT NOT NULL DEFAULT 'subj_high_level',
    exam_date     TEXT NOT NULL,
    weekly_days   INTEGER NOT NULL CHECK (weekly_days BETWEEN 0 AND 7),
    daily_minutes INTEGER NOT NULL CHECK (daily_minutes > 0)
);

-- 学习记录（阅读位置 + 答题状态）
CREATE TABLE user_records (
    record_id    TEXT PRIMARY KEY,
    user_id      TEXT NOT NULL,
    target_type  TEXT NOT NULL,  -- point / question / paper
    target_id    TEXT NOT NULL,
    status       TEXT NOT NULL,  -- not_start / doing / done
    last_position TEXT,
    answer_time_ms INTEGER,     -- 答题耗时，仅 question 类型记录
    updated_at   TEXT NOT NULL,
    synced       INTEGER DEFAULT 0  -- 是否已同步到云端
);

-- 错题本
CREATE TABLE wrong_book (
    wrong_id      TEXT PRIMARY KEY,
    user_id       TEXT NOT NULL,
    question_id   TEXT NOT NULL,
    wrong_count   INTEGER DEFAULT 1,
    right_streak  INTEGER DEFAULT 0,
    priority      TEXT NOT NULL,  -- high / normal / archived
    last_wrong_at TEXT NOT NULL,
    next_review_at TEXT,          -- 艾宾浩斯下一次复习时间（本地引擎计算）
    synced        INTEGER DEFAULT 0
);

-- 掌握度
CREATE TABLE mastery_scores (
    user_id     TEXT NOT NULL,
    point_id    TEXT NOT NULL,
    score       INTEGER NOT NULL,  -- 0-100
    updated_at  TEXT NOT NULL,
    synced      INTEGER DEFAULT 0,
    PRIMARY KEY (user_id, point_id)
);

-- 每日计划任务
CREATE TABLE daily_tasks (
    task_id         TEXT PRIMARY KEY,
    user_id         TEXT NOT NULL,
    task_date       TEXT NOT NULL,
    task_type       TEXT NOT NULL,  -- learn / practice / wrong_review
    target_id       TEXT,
    target_count    INTEGER,
    completed_count INTEGER DEFAULT 0,
    status          TEXT NOT NULL,  -- pending / doing / done / skipped
    synced          INTEGER DEFAULT 0
);
```

每个表都带 `synced` 字段，标记该行是否已同步到服务端。离线时 `synced = 0` 的行在网络恢复后由后台 Sync 任务逐条上传。

### 1.8 PDF → 结构化数据转换流程

PDF 内容经 AI 解析后直接输出 **MD 文件**；元数据由运营人员在 JSON 文件中手工维护或通过脚本批量生成。

```
PDF 文件（软考知识点集合）
        │
        ▼
┌─────────────────────────┐
│ ①  文本提取              │
│  Python pdfplumber      │  → 保留表格结构，输出带坐标的文本块
└──────────┬──────────────┘
           ▼
┌─────────────────────────┐
│ ②  AI 结构化解析         │
│  Claude API             │  → 按本文档 2.1.3 语法规范输出 MD
│                          │    含 ==重点==、:::type 块、表格
└──────────┬──────────────┘
           ▼
┌─────────────────────────┐
│ ③  人工审核 + 编辑       │
│  任意 Markdown 编辑器    │  → 审校内容准确性和 MD 语法
│                          │    标注 ==红字重点==
│                          │    补充记忆口诀（:::tip 块）
└──────────┬──────────────┘
           ▼
┌─────────────────────────┐
│ ④  元数据录入            │
│  JSON 手工 / 脚本批量     │  → 建立 kp_xxx.json 元数据文件
│                          │    填写 difficulty、关联关系等
└──────────┬──────────────┘
           ▼
┌─────────────────────────┐
│ ⑤  版本管理 + 打包       │
│  Git + CDN              │  → JSON + MD 入 Git 仓库
│                          │    增量打包 → 上传 CDN
└─────────────────────────┘
```

**为什么要 AI 辅助？**
- 软考知识点含大量表格，人工从 PDF 录入易错且慢
- LLM 可直接识别表格行列结构并输出对应的 `table` 块 JSON
- 人工只需审核校对，不需要从零打字或调格式
- 单科目预计 200+ 张卡片，纯人工效率太低

### 1.9 iOS App 读写架构

#### 1.9.1 读取静态内容（JSON 元数据 + MD 正文）

```
           ┌─ 网络可用 ──→ CDN 检查更新 ──→ 下载增量 JSON + MD ──→ 写入本地沙盒
           │
App 启动 ──┤
           │
           └─ 离线 ──────→ 读取本地沙盒缓存
                                │
                    ┌───────────┴───────────┐
                    ▼                       ▼
           JSONDecoder              MD 解析器
           → KnowledgeCardMeta      → 渲染为 SwiftUI View
           （元数据、关联关系）        （正文内容）
```

**读取优先级**：本地沙盒缓存 > Bundle 内置默认数据。JSON 和 MD 各自独立缓存，任一文件更新只影响对应部分。

**关键代码路径示意**：

```
CardRepository
  ├── localCacheURL (Documents/cache/cards/)
  ├── bundleURL (Bundle.main)
  ├── loadMeta(pointId:) → 本地 JSON ?? Bundle ?? nil
  │     └── JSONDecoder → KnowledgeCardMeta (Codable)
  ├── loadContent(fileName:) → 本地 MD ?? Bundle ?? nil
  │     └── MarkdownParser → CardContent → SwiftUI View
  ├── checkUpdate() → 比对 CDN 版本号 → 下载变更的 JSON/MD
  └── deleteOldCache() → 清理过期缓存

MarkdownRenderer (SwiftUI)
  ├── 标准 Markdown 元素（由解析库处理）
  │     ├── 标题、段落、列表、图片、代码块
  │     └── 表格 → 按列数自动选择三策略渲染
  │
  └── 自定义扩展语法（客户端拦截处理）
        ├── ==重点文字==  → 红色加粗渲染（受 toggle 控制）
        └── :::type 块    → HighlightView（tip/warning/key 风格）
```

#### 1.9.2 写入动态数据（学习记录、错题）

```
用户操作（答题 / 翻页）
        │
        ▼
GRDB 本地 SQLite 即时写入 (< 1ms，不阻塞 UI)
        │
        ▼
标记 synced = 0
        │
        ▼
┌─ 有网络 ──→ BackgroundSync 任务
│              ├── 读取 synced = 0 的行
│              ├── 逐条 POST 到服务端
│              ├── 服务端返回成功 → 标记 synced = 1
│              └── 服务端返回纠偏数据（艾宾浩斯时间校准）
│
└─ 无网络 ──→ 静默等待，NetworkReachability 监听恢复后自动触发
```

**冲突解决策略**（多端离线后同步）：
- 以 `updated_at` 最后修改时间为准，`last-write-wins`
- 错题记录按 `wrong_id` 去重，合并 `wrong_count`（取最大值）
- 服务端对艾宾浩斯 `next_review_at` 做纠偏覆盖（服务端为准）

### 1.10 技术栈选型

#### iOS 客户端

| 层 | 技术 | 用途 | 选型理由 |
| :--- | :--- | :--- | :--- |
| UI | SwiftUI | 声明式 UI，卡片渲染 | 原生性能，Swift 生态一致 |
| 数据模型（元数据） | Swift `Codable` | JSON ↔ 模型双向转换 | 零依赖，原生支持 |
| Markdown 渲染 | [swift-markdown](https://github.com/apple/swift-markdown) | MD 正文解析 + 自定义扩展 | Apple 官方库，轻量（~200KB），支持 Visitor 模式扩展自定义语法 |
| 本地数据库 | [GRDB](https://github.com/groue/GRDB.swift) | SQLite 封装 | 比 Core Data 轻量，比 FMDB 更 Swift 化，支持 Combine 响应式监听 |
| 网络 | `URLSession` + `Async/Await` | HTTP 请求、文件下载 | 原生，无需 Alamofire |
| 离线包管理 | `FileManager` + `URLSession` | JSON + MD 文件下载、缓存、版本比对 | 原生能力即可满足 |
| 音频 TTS | `AVSpeechSynthesizer` | 知识卡片语音朗读 | iOS 原生 TTS，无需第三方 |
| 表格渲染 | SwiftUI `Grid` / `ScrollView` / `UIViewRepresentable(WebView)` | 表格三策略渲染 | 复杂表格可降级用 WebView + CSS，其余原生 |
| 小组件 | `WidgetKit` | 桌面小组件 | iOS 原生 |
| 推送 | `UserNotifications` + APNs | 本地 + 远程通知 | iOS 原生 |
| IAP | `StoreKit 2` | 买断制内购 | Swift 原生，现代 API |

#### 服务端

| 层 | 技术 | 用途 |
| :--- | :--- | :--- |
| API 服务 | Go / Node.js | REST API，轻量高性能 |
| 数据库 | PostgreSQL | 用户账号、学习记录、错题同步 |
| 缓存 | Redis | 热点数据缓存、Session |
| 文件分发 | 阿里云 OSS / AWS S3 + CDN | JSON 卡片文件、离线包分发 |
| IAP 验证 | Apple Server API | 购买凭证服务端二次验证 |

## 2. 数据结构设计

### 2.1 知识卡片 (Knowledge Card)

> 对应 PRD `12.2 知识点`。PRD 中保留精简字段定义，本文档为完整数据设计。

知识卡片采用 **JSON（元数据）+ Markdown（正文）** 双文件方案。JSON 供程序查询、筛选、生成任务；Markdown 供人工编辑和客户端渲染。

#### 2.1.1 卡片元数据（JSON 文件）

| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| point_id | string | 知识点唯一标识 |
| subject_id | string | 科目 ID |
| chapter_id | string | 所属章节 ID |
| section_id | string | 所属子章节 ID，如 `"sec_01_01"`（V1.4 新增）。与 `chapter_meta.sections[].section_id` 对应 |
| title | string | 卡片标题（单行，≤30 字） |
| card_type | enum | 卡片类型：`concept`（概念）/ `process`（流程）/ `comparison`（对比）/ `formula`（公式）/ `definition`（定义） |
| difficulty | int | 内容难度 1-3，与题目难度体系（PRD 11.2）对齐 |
| estimated_read_seconds | int | 预估阅读时长（秒），用于微任务包生成时做时间预算 |
| has_key_content | bool | 是否包含 ==重点标注==，客户端据此决定是否显示"显示/隐藏重点"切换按钮 |
| is_free | bool | 是否免费用户可见 |
| sort_no | int | 章节内排序序号 |
| content_file | string | 对应 MD 文件名，如 `"kp_ch04_001.md"`，客户端据此加载正文 |
| tags | string[] | 关键词标签数组，用于分类检索和智能推荐。如 `["信息","信息化","信息系统"]`（V1.3 新增） |
| key_points | string | 高频考点内容（Markdown 格式）。渲染为正文下方的独立考点卡片，视觉上与正文阅读区分开。支持粗体、列表等基础 MD 语法（V1.5 新增） |
| mnemonics | string | 记忆口诀内容（Markdown 格式）。渲染为考点卡片下方的独立口诀卡片。支持粗体、列表等基础 MD 语法（V1.5 新增） |

#### 2.1.2 Markdown 正文格式规范

知识卡片正文采用标准 Markdown 格式，辅以少量自定义扩展语法。Markdown 作为内容源格式，人工可直接编辑；客户端通过解析器渲染为原生 SwiftUI 视图。

**选择 Markdown 的理由**：
- 任意编辑器可打开，运营人员零学习成本
- Git diff 可读，修改一个字不会产生 JSON 结构噪音
- 标准语法覆盖 90% 内容场景（标题、段落、列表、表格、代码）
- 少量自定义扩展（`==重点==`、`:::type`）覆盖特殊需求

#### 2.1.3 Markdown 自定义语法扩展

##### 标准 Markdown 元素

以下元素遵循 CommonMark 规范，由 Markdown 解析库直接处理：

| 元素 | 语法 | 说明 |
| :--- | :--- | :--- |
| 标题 | `#### 概述`、`#### 关键要点` | 从 level 3 开始（level 1 留给卡片标题 meta，level 2 预留给未来扩展） |
| 段落 | 普通文本 | 连续非空行自动聚合为段落 |
| 有序列表 | `1. 第一项` | 数字 + 点 + 空格 |
| 无序列表 | `- 项目`、`* 项目` | 短横或星号 |
| 图片 | `![替代文本](url)` | 标准图片语法 |
| 代码 | `` `inline` `` 或 ` ```block``` ` | 行内代码和代码块 |
| 粗体/斜体 | `**粗体**`、`*斜体*` | 标准内联格式 |

##### ==重点标注==（key_point）★

使用 `==文字==` 标记段落内的重点内容（红字）。这是 Markdown 社区已有的高亮扩展语法（Obsidian、Typora 等均支持），无需额外学习。

```markdown
制定项目章程是编写一份正式批准项目并==授权项目经理在项目活动中使用组织资源==的文件的过程。==章程批准后项目才算正式立项。==
```

**解析规则**：
- `==` 之间的文字标记为 `key_point` 样式
- 可与其他内联格式叠加：`==**粗体重点**==`
- 每个 `==..==` 对在段落内独立计算起止位置

##### :::type 高亮块（highlight）

使用 `:::` 围栏语法创建带视觉风格的独立高亮块，用于突出核心考点、记忆口诀、易错提示。

```
:::tip
**记忆口诀**：章程给名分，计划给方向。谁签发章程谁就是项目发起人。
:::

:::warning
**易错提示**：项目章程由**发起人**签发，不是项目经理自行批准。
:::

:::key
制定项目章程是整合管理的起点，属于**启动过程组**。
:::
```

| type | 视觉风格 | 用途 |
| :--- | :--- | :--- |
| `tip` | 绿色边框 + 绿色图标 | 记忆口诀、学习技巧 |
| `warning` | 橙色边框 + 橙色图标 | 易错提示、常见误区 |
| `key` | 蓝色边框 + 蓝色图标 | 核心考点、关键结论 |

**解析规则**：
- `:::` 后紧跟的单词为 type，同一行其余内容忽略
- 开始 `:::` 和结束 `:::` 之间的内容为块正文，支持内联 Markdown（粗体、重点标注等）
- 不支持嵌套

##### 表格 + 渲染提示注释

表格使用标准 Markdown 表格语法。如需指定移动端渲染策略，在表格上方一行添加 HTML 注释携带元数据。标准 Markdown 解析器会忽略该注释，客户端解析器会提取渲染参数。

```
<!-- render:native caption:制定项目章程的输入输出 -->
| 输入 | 工具与技术 | 输出 |
|:---|:---|:---|
| 项目章程 | 专家判断 | 项目章程（签发） |
| 商业文件 | 数据收集 | 假设日志 |
```

**注释字段**（空格分隔，均为可选）：

| 字段 | 说明 | 默认值 |
| :--- | :--- | :--- |
| `render:native\|scroll\|card_fold` | 渲染策略，不指定则由客户端根据列数自动判断 | 自动判断 |
| `first_col_fixed` | 横向滑动时首列固定 | false |
| `col_weights:3,2,3,1` | 列宽权重 | 均分 |
| `caption:表4-1` | 表格标题 | 无 |

**渲染策略自动判断规则**（未显式指定 render 时）：

| 列数 | 自动策略 | 渲染方式 |
| :--- | :--- | :--- |
| ≤3 | `native` | 原生表格，斑马纹 + 表头底色，字号比正文小 1pt |
| 4-5 | `scroll` | 横向滑动容器，右侧渐变阴影提示，双击进入全屏横屏视图 |
| ≥6 | `card_fold` | 每行折叠为独立卡片，收起显示首列摘要，展开显示完整 key-value 对 |

> **运营规范**：单表不超过 5 列。超过 5 列需在录入阶段拆分为多张子表，或改用列表/流程图替代。

#### 2.1.7 重点内容切换功能

客户端在知识卡片阅读页提供一个切换按钮，控制 `==重点标注==` 内容的显示与隐藏。

**交互规则**：

| 状态 | 按钮图标 | 重点文字渲染 |
| :--- | :--- | :--- |
| 显示模式（默认） | 眼睛睁开 | 红色加粗，与正文形成高对比 |
| 隐藏模式 | 眼睛闭合/斜线 | 隐藏或显示为占位符（如 `···`），保持句子结构可读 |

**按钮可见性**：
- 元数据 `has_key_content = true` → 显示切换按钮
- `has_key_content = false` 或卡片无任何 `==..==` 标注 → 不显示按钮

**全局持久化**：
- 切换状态记录在本地 `UserDefaults`，key 为 `key_point_visible`
- 所有卡片的重点内容统一受此开关控制，不需要每张卡片单独设置

**渲染逻辑**（客户端 `MarkdownRenderer` 实现）：

```
解析 MD 文本 → 识别 ==..== 标记
                   │
       ┌───────────┴───────────┐
       ▼                       ▼
  显示模式                  隐藏模式
  红色 + 粗体               opacity: 0
  正常显示                  占位符 "···"（可选）
```

#### 2.1.8 Markdown → SwiftUI 渲染规则

客户端使用 `swift-markdown` 解析 MD 文件，通过自定义 `Visitor` 处理扩展语法，将 Markdown AST 映射为 SwiftUI 视图。

**标准元素映射**：

| Markdown | SwiftUI 视图 |
| :--- | :--- |
| `### 标题` (level 3) | `Text().font(.title3).fontWeight(.bold)` |
| `#### 标题` (level 4) | `Text().font(.body).fontWeight(.semibold)` |
| 段落 | `Text().font(.body)` + 内联样式处理 |
| `- 列表` / `1. 列表` | `VStack` + 每项 `HStack { Circle(); Text() }` |
| `**粗体**` | `Text().fontWeight(.bold)` |
| `![alt](url)` | `AsyncImage` + 点击全屏预览 |
| `` `code` `` | `Text().font(.system(.caption, design: .monospaced)).background(Color.gray.opacity(0.1))` |

**自定义语法映射**：

| Markdown 扩展 | SwiftUI 视图 |
| :--- | :--- |
| `==重点==` | `Text().foregroundColor(.red).fontWeight(.bold)` → 受 `key_point_visible` 全局状态控制 |
| `:::tip ... :::` | `HighlightView(style: .tip)` → 绿色边框 + 💡图标 |
| `:::warning ... :::` | `HighlightView(style: .warning)` → 橙色边框 + ⚠️图标 |
| `:::key ... :::` | `HighlightView(style: .key)` → 蓝色边框 + 🔑图标 |
| `<!-- render:... -->` + 表格 | `TableView(strategy:)`，根据 render_hint 或列数选择 native/scroll/card_fold 策略 |

**公式支持**（可选，P2）：
- 行内公式 `$E=mc^2$` → `MathView(latex:)` 
- 块级公式 `$$...$$` → `MathView(latex:, display: true)`
- 如 V1 不启用公式渲染，客户端将 `$..$` 按普通文本处理
